# Listen Server Multiplayer Audit

본 문서는 `MyHackSlash`의 리슨 서버(Listen Server) 멀티플레이 전환 상태와 남은 작업을 추적합니다. 기준은 Unreal Engine 5.5, PIE Listen Server, 서버 권한(authority) 중심 게임플레이, 클라이언트 로컬 UI/입력 분리입니다.

데디케이티드 서버(Dedicated Server)는 추후 패키징/운영 단계의 확장 목표입니다. 현재 Epic Games Launcher 배포판 UE 5.5에서는 서버 타겟 빌드가 제한되므로, 당장은 `MyHackSlashEditor Win64 Development` 빌드와 PIE Listen Server 검증을 기준으로 판단합니다.

## 2. 최근 검증 결과

2026-05-18 PIE Listen Server 로그 기준:

| 항목 | 결과 |
| :--- | :--- |
| `Error/Warning/Ensure/Crash/LogScript/LogNet` | 0건 |
| `COIN_PICKUP_REJECT_SERVER` | 0건 |
| `HPlayerState_1` 코인 획득 승인 | 39건 |
| `HPlayerState_0` 코인 획득 승인 | 1건 |
| `PickupActive=1` 활성화 multicast | 서버 28건 / 클라이언트 28건 |

코인 관련 결론:

- 코인 획득은 서버 overlap만 권위 처리합니다.
- 클라이언트 pickup RPC 경로는 제거했습니다.
- `AHCoin` 활성화 multicast에 서버 `Location`과 팝콘 초기 `Velocity`를 포함해 클라이언트 표시 위치/궤적 불일치를 줄였습니다.
- 획득 후 즉시 풀에 넣지 않고 짧게 지연해, 비활성 multicast가 클라이언트에 도달하기 전에 같은 액터가 재활성화되는 문제를 완화했습니다.
- 코인 PIE 검증용 `PIE_CHECK` 로그는 검증 완료 후 코드에서 제거했습니다.

젬 장착 관련 결론:

- 메인 젬 장착/해제와 보조 젬 장착/해제는 서버 승인 및 OnRep 갱신까지 정상 확인했습니다.
- 클라이언트 드래그/드롭에서 장착 상태, 인벤토리 수량, Ability 부여 상태가 일치하는 것을 확인했습니다.
- 젬 장착 PIE 검증용 `PIE_CHECK` 로그는 검증 완료 후 코드에서 제거했습니다.

## 3. 남은 작업 우선순위

| 우선순위 | 작업 | 이유 | 검증 기준 |
| :--- | :--- | :--- | :--- |
| 1 | 플레이어 1 레벨업 선택창 미표시 수정 | 클라이언트 보상 선택 UI가 뜨지 않으면 멀티플레이 진행이 막힘 | 플레이어 0/1 모두 레벨업 시 본인 선택창 표시, 선택 결과 서버 적용 |
| 2 | GameplayCue 기반 투사체 폭발 VFX/SFX 정리 | 데미지 판정은 서버화됐지만 연출 중복/누락 가능성 남음 | 서버 데미지는 1회, 모든 클라이언트에 폭발/VFX/SFX가 1회 표시 |
| 3 | 웨이브 결과/상점 UI pause 정책 분리 | 플레이어별 UI와 서버 진행 상태가 섞일 가능성 | 한 플레이어 UI 조작이 다른 플레이어 입력/진행을 비의도적으로 멈추지 않음 |
| 4 | 공유 경험치/멀티플레이 레벨업 선택 구현 | 정책만 정해졌고 실제 흐름은 남아 있음 | 모든 플레이어 선택 UI 표시, 10초 후 미선택 자동 선택, 모든 선택 후 게임 재개 |
| 5 | 다중 플레이어 기준 맵 타일 유지 | 멀리 떨어진 플레이어 주변 타일/네비 부족 가능 | 두 플레이어가 떨어져도 각자 주변 타일과 네비 유지 |
| 6 | 래그돌 물리 복제 품질 보정 | 게임플레이 정확성보다 연출 품질 이슈 | 사망 연출이 호스트/클라이언트에서 과도하게 튀거나 다르게 보이지 않음 |

권장 시작점은 `플레이어 1 레벨업 선택창 미표시 수정`입니다. 먼저 서버에서 해당 플레이어의 레벨업 이벤트와 `ClientShowSelectAbilityPopup()` 호출 여부를 확인한 뒤, 보상 선택 UI의 owning player 및 옵션 복제 흐름을 정리합니다.

## 4. 유지해야 할 멀티플레이 원칙

- 게임플레이 결과가 있는 동작은 서버에서 판정합니다.
- 클라이언트 UI는 복제 상태를 읽고, 변경은 서버 RPC로 요청합니다.
- 월드 아이템은 서버 overlap에서 효과를 적용하고, 클라이언트는 actor/movement replication 및 multicast로 표시만 맞춥니다.
- 로컬 UI, 입력 모드, 설정 저장, 사운드는 클라이언트 전용으로 처리합니다.
- 웨이브처럼 모두가 알아야 하는 상태는 GameState, 플레이어별 상태는 PlayerState/PlayerController 쪽에 둡니다.
- WorldSubsystem/GameInstanceSubsystem은 편하지만 네트워크 복제 주체가 아니므로, 복제 상태 저장소로 쓰지 않습니다.

## 5. 검증 기준

### 기본 PIE

- `MyHackSlashEditor Win64 Development` 빌드 통과
- PIE `Number of Players = 2`, `Net Mode = Play As Listen Server`
- 호스트와 클라이언트가 각각 본인 HUD만 생성
- 서버 권한 게임플레이와 로컬 UI/사운드 분리

### 전투/아이템

- 클라이언트 공격도 서버에서만 데미지 적용
- 자동 슬롯 스킬 중복 발동 없음
- 투사체 데미지 1회, 연출은 모든 클라이언트에 정상 표시
- 몬스터 사망과 코인 드랍 중복 없음
- 코인 획득 reject 없음, 서버/클라이언트 활성화 multicast 수 일치

### UI/보상

- HP/Gold/Exp/Hunger가 각 클라이언트에서 올바르게 갱신
- 한 클라이언트 설정창이 다른 플레이어 게임을 멈추지 않음
- 상점 구매와 보상 선택은 서버 검증 후 적용
- 젬 드래그/드롭은 메인/보조 젬 장착/해제 후 서버 상태, 인벤토리, 양쪽 UI 표시가 일치
- 멀티플레이 레벨업 선택은 모든 플레이어에게 표시되고 타임아웃 자동 선택 처리

## 6. 장기 참고

- 데디케이티드 서버 타겟(`MyHackSlashServer.Target.cs`)은 유지하되, 실제 빌드는 서버 타겟을 지원하는 UE 소스 빌드 설치본이 필요합니다.
- 리슨 서버 단계에서는 서버 타겟 빌드보다 PIE 멀티플레이에서 권한과 복제 흐름이 올바른지가 우선입니다.
- 복제 액터 풀링은 hidden/collision/tick/dormancy/movement/owner/instigator를 함께 리셋해야 합니다.
